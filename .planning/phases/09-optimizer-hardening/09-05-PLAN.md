### Wave 3: Flamegraph Profiling Notes + Optimization
**Files:**
- `apps/quantedge_core/PERF_NOTES.md` — document profiling findings

**Details:**
Document known hotspots and optimization decisions:

1. **Bar grouping**: Current `BTreeMap<(NaiveDate, NaiveTime), Vec<&SimBar>>` grouping is O(N log N). Pre-sorting bars once is crucial.

2. **Strike matching**: Linear scan per time slot. For large strike ranges, consider pre-building a `HashMap<(date, time, option_type, strike_offset), &SimBar>` index.

3. **Metrics computation**: Single-pass where possible. Daily aggregation uses `BTreeMap<NaiveDate, f64>` which is efficient for sorted access.

4. **Memory**: Each SimBar is ~80 bytes. 4yr BankNifty (~8000 bars) ≈ 640KB. Multiple copies during optimizer sweep via Rayon is acceptable (each thread clones bars reference via `Arc`).

5. **Rayon pool**: Default thread pool = num_cpus. For 720 combos on 8-core machine: ~90 combos per core. Each combo takes ~1ms → total ~90ms per core → ~90ms wall time. Well under 3-minute target.

6. **Suggested optimization**: If benchmarks show regression, wrap bars in `Arc<Vec<SimBar>>` and pass shared reference to Rayon tasks instead of cloning.
